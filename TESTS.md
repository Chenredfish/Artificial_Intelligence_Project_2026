# AI Battle 測試計畫

> 使用 AI Battle 批次測試功能執行。先手一律設「隨機」以消除先手偏差。
> 短版本快速確認方向；長版本才有統計意義（30 局以上 p < 0.05 可信）。

---

## 1. 冒煙測試（功能是否正常）

驗證所有功能可以正常跑完，不崩潰。

| 項目 | 設定 |
|------|------|
| AB / UV | Minimax, d=5, NMP✓, LMR✓ |
| 先手 | 隨機 |
| 時限 | 10s |

**短版本：** 3 局  
**長版本：** 10 局

**預期結果：** 所有局正常結束，狀態列顯示「完成 N 局」，勝負分布無明顯異常（兩隊都有機會獲勝）。

---

## 2. 平行模式相容性測試（新修正驗證）

確認平行模式在有時限的情況下可以正常運作（舊版平行僅在無時限時觸發）。

| 項目 | 短版本設定 | 長版本設定 |
|------|-----------|-----------|
| AB | Minimax, d=50, 平行✓ | 同左 |
| UV | Minimax, d=50, 平行✓ | 同左 |
| 時限 | 10s | 30s |
| 局數 | 5 局 | 20 局 |

**預期結果：**
- 每局耗時應接近設定時限（10s 時限每局約 10–12s），不會跑超過數倍時間
- 兩隊平行都啟動時，實際 CPU 使用率應明顯上升（可從工作管理員確認）
- 勝負分布接近 50/50（對稱設定）

---

## 3. 平行 vs 循序強度測試（同時限）

確認平行在相同時限下是否比循序更強。  
核心問題：同樣 60s，平行能搜到更深還是 TT 共享損失更大？

| 項目 | AB（對照組） | UV（實驗組） |
|------|------------|------------|
| 策略 | Minimax | Minimax |
| 深度 | 50 | 50 |
| 平行 | 否 | 是 |
| NMP / LMR | ✓ | ✓ |
| 時限 | 60s | 60s |

**短版本：** 10 局  
**長版本：** 50 局

**預期結果：**
- 若平行較強：UV 勝率 > 55%
- 若差異不顯著：接近 50/50（表示 TT 共享損失大致抵銷了多核優勢）
- 若循序較強：考慮比賽時關閉平行

---

## 4. LMR 強度測試

驗證 LMR 是否真的讓 AI 更強（而非只是加速）。

| 項目 | AB（無 LMR） | UV（有 LMR） |
|------|------------|------------|
| 策略 | Minimax | Minimax |
| 深度 | 50 | 50 |
| LMR | 否 | 是 |
| NMP | ✓ | ✓ |
| 時限 | 60s | 60s |

**短版本：** 10 局  
**長版本：** 50 局

**預期結果：**
- UV（有 LMR）應有明顯勝率優勢（> 55%），因為相同時限內 LMR 讓 ID 多走幾層
- 若 UV 勝率 < 50%：LMR 的減枝損失了關鍵分支，需要考慮調整 `min-d` 或 `start` 參數

## 6. LMR 參數調整測試

找出最佳的 `min-d` 和 `start` 組合。建議每次只改一個變數。

### 6A — 調整 min-d（啟用 LMR 的最低深度）

| 組合 | min-d | start | 預期 |
|------|-------|-------|------|
| 保守 | 4 | 3 | 減少 LMR 誤剪，較安全 |
| 預設 | 3 | 3 | 目前設定 |
| 激進 | 2 | 3 | 更大幅縮減，可能損失精度 |

每組設定：AB=對照（min-d=3），UV=實驗，Minimax d=50，60s，**20 局**。  
**長版本：** 每組 50 局。

**預期結果：** min-d=3 或 4 應優於 min-d=2；若 min-d=4 也更好，更新預設值。

### 6B — 調整 start（從第幾個走法開始縮減）

| 組合 | min-d | start |
|------|-------|-------|
| 保守 | 3 | 5 |
| 預設 | 3 | 3 |
| 激進 | 3 | 1 |

設定同上。  
**預期結果：** start=3 或 5 應優於 start=1（太激進縮減前幾個好走法）。

## 7. 深度影響測試（無時限固定深度）

了解在固定深度下，多深才真正有強度差異。

| 項目 | AB | UV |
|------|----|----|
| 策略 | Minimax | Minimax |
| 深度 | 5 | 7 |
| 時限 | 0（無時限） | 0（無時限） |
| NMP / LMR | ✓ | ✓ |

**短版本：** 10 局（注意：無時限每局可能需要 1–5 分鐘）  
**長版本：** 30 局

也可測試 d=7 vs d=9，依此類推。

**預期結果：**
- depth=7 應明顯強於 depth=5（UV 勝率 > 60%）
- 若差距很小：表示 depth=5 已接近飽和，不需要搜更深

**測試 B：Minimax vs Greedy**

|------|----|----|
| 策略 | Minimax d=5 | Greedy |
| 時限 | 10s | — |

**短版本：** 10 局，**預期：** AB 勝率 > 65%

---

## 9. 評估函式權重調整測試

目前權重：A/U=10, B/V=8, c/d/w/x=4, e/f/y/z=3（僅 AI 啟發，非計分）。  
調整一個參數，其他不變，跑 30+ 局比較。

**建議先測：提升小棋重要性（c/d 從 4 調到 3，貼近實際計分比例）**

| 項目 | AB（原始權重） | UV（調整後） |
|------|--------------|------------|
| 策略 | Minimax d=50 | Minimax d=50 |
| 時限 | 30s | 30s |

**長版本：** 50 局（權重調整影響細微，需要更多局數）

**預期結果：**
- 若 UV 勝率 > 55%：調整後更好，更新 `EVAL_PIECE_WEIGHTS`
- 若接近 50/50：差異不顯著，保持原設定

---

## 測試執行順序建議

```
優先（功能驗證）：  1 → 2
強度排名確認：      8
新功能效果：        3 → 4 → 5
參數調整：          6A → 6B → 9
深度研究：          7（耗時，選擇性執行）
```

## 結果判讀標準

| 勝率差距 | 結論 |
|---------|------|
| < ±5%   | 統計雜訊，無顯著差異 |
| ±5–15%  | 小幅優勢，值得保留但非關鍵 |
| > ±15%  | 顯著優勢，確認該設定更強 |
| > ±30%  | 壓倒性優勢，幾乎可確定結論 |

> 30 局樣本下，5% 勝率差異約在統計誤差邊緣；50 局以上才可信。
